\# CephasOps – Compact UI Ruleset



All frontend pages MUST follow a compact layout style similar to the OLD CephasOps UI design.



GENERAL RULES

\- Reduce visual spacing across the entire page.

\- Use compact spacing for grids (gap-2 or gap-3, never gap-6 or higher).

\- Section padding must be small (p-3 or p-4, not p-6 or p-8).

\- Input elements should be compact with reduced height (around 34–38px).

\- Do NOT use oversized inputs or large white spacing.

\- All headers should be small (text-sm or text-base), bold, clean.

\- Keep forms dense and efficient, similar to enterprise systems (not mobile-style).



FORM INPUT RULES

\- Input height must be compact.

\- Inputs must not have excessive padding.

\- Avoid large text fields unless necessary.

\- Labels must be small (text-xs or text-sm).



LAYOUT RULES

\- Follow the OLD CephasOps layout structure:

&nbsp; - Tight grouping of related fields.

&nbsp; - Horizontal grouping of 2–4 inputs per row.

&nbsp; - Use compact cards with light borders for each section.

&nbsp; - Remove unnecessary vertical white space.



AVOID

\- Large padding (p-6, p-8).

\- Big input boxes (default Radix/Vite tall input styles).

\- Excessive vertical spacing.

\- Oversized form groups.



GOAL

Cursor must generate UI that feels like:

\- Enterprise-grade internal tool

\- Fast, clean, compact

\- Similar to the OLD CephasOps Activation/Modification form



If Cursor generates a UI that looks too big, too spacious, or oversized,

IMMEDIATELY adjust it to be compact before returning the final answer.
