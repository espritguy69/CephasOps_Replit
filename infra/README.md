🚀 CephasOps Infrastructure Overview

This folder defines how CephasOps is deployed, scaled, monitored, and maintained across all environments:

Local Development

Staging / UAT

Production

Infrastructure is designed to be:

Stateless at the application layer

Repeatable (IaC)

Observable (metrics + logs)

Secure (RBAC, isolation, secrets management)

Multi-company aware (backend logic, not infra-level)

All persistent business state lives in:

PostgreSQL

Object storage (S3-compatible storage)

Redis (optional caching)

Persistent logs (ELK or vendor)

Background job state tables (in Postgres)

📁 Folder Structure
infra/
  README.md            ← This file
  k8s/                 ← Kubernetes manifests (deployment, service, ingress)
  terraform/           ← Cloud infrastructure (IaC)
  monitoring/          ← Prometheus, Grafana, alert rules
  (docker/ and ci-cd/  ← Not yet in repo; local dev uses host-run backend/frontend, CI is .github/workflows)

🧱 1. Architecture Summary

The CephasOps platform runs as a set of backend and frontend services with supporting infrastructure:

Backend

.NET API

Background Jobs (Hangfire or Worker Service)

Parser Workers (email ingestion, snapshot cleanup)

Scheduler Workers (nightly cron)

Frontend

Admin Web Portal (/frontend)

SI Mobile App (/frontend-si)

Hosted via:

Static hosting (S3 + CloudFront)

or via NGINX in Kubernetes

Data Layer

PostgreSQL (primary system of record)

Redis (optional)

Object storage (S3/MinIO)

Observability

Prometheus (metrics)

Grafana (dashboards)

Loki/ELK (logs)

Alertmanager (alerts)

🏗️ 2. Local Development

Local dev currently uses host-run services (see repo root scripts: start-backend.ps1, start-frontend.ps1, start-all.ps1). No docker-compose is in the repo yet.

A future infra/docker/ could include:

backend-api

scheduler-worker

parser-worker

postgres

redis

minio

pgadmin or adminer

Run:

docker compose up --build


Frontend is run separately:

cd frontend
npm run dev

☁️ 3. Staging / Production Deployment

CephasOps uses Kubernetes for production.

K8s manifests live under:

infra/k8s/

Includes:

Deployments (api, background-jobs, parser-worker)

Services

Ingress (Nginx / Traefik)

ConfigMaps

Secrets

HorizontalPodAutoscaler (HPA)

CronJobs (snapshot cleanup, parser sync, nightly rebuilds)

Backend Deployment:

api-deployment.yaml

scheduler-worker-deployment.yaml

parser-worker-deployment.yaml

Frontend Deployment:

admin-frontend-deployment.yaml

si-app-deployment.yaml (optional if hosting as PWA/SPA)

Configured to use:

Rolling deployments

Restart policies

Liveness & readiness probes

Example probe:

livenessProbe:
  httpGet:
    path: /health/live
    port: 8080

🛠️ 4. Infrastructure as Code (IaC)

All cloud infra is provisioned via Terraform:

infra/terraform/


Typical stacks include:

VPC / Networking

Kubernetes cluster (EKS, AKS, or GKE)

RDS PostgreSQL

Redis (Elasticache / MemoryStore)

S3 Buckets (file storage)

CloudFront (admin frontend)

Load balancers

Monitoring stack

State is stored via remote Terraform state (S3/GCS/Azure).

🧹 10. Principles
✔ Stateless applications

Everything horizontal-scalable.

✔ Idempotent background jobs

Snapshot cleanup, parser ingestion, P&L rebuild.

✔ Isolation by environment

No shared resources between environments.

✔ Environment parity

Dev → Staging → Production match in structure.

✔ Scalability

Everything auto-scalable except stateful services.
