# EF InitialBaseline + Current Schema — Strategy

**Status:** Strategy only; **not executed**. Use when the team decides to adopt an InitialBaseline for new environments while keeping the legacy chain for existing ones.

**Related:** `docs/operations/EF_REBASELINE_PLAN.md`, `docs/operations/EF_MIGRATION_GRAPH_INTEGRITY_AUDIT.md`, `docs/operations/EF_MIGRATION_GRAPH_INTEGRITY_DECISION.md`, `docs/operations/EF_BASELINE_CUTOVER_DECISION_MATRIX.md`.

---

## 1. When CephasOps should stop extending the legacy chain

- **Do not stop** until the team has:
  - Agreed to a two-track model (legacy for existing DBs, baseline for new).
  - Created and validated the InitialBaseline migration and snapshot.
  - Updated runbook, CI, validator, and classifier for the chosen model.
- **Optional stop:** Once InitialBaseline exists and is validated, **new** migrations can be added **either** to the legacy chain (so existing DBs get them via `database update`) **or** to the baseline track (so only new/greenfield DBs get them from the new chain). The decision of “where to add the next migration” is a policy choice: usually “add to legacy” until legacy is frozen, then “add only to baseline.”
- **Recommended:** Continue extending the legacy chain for all new migrations until the cutover criteria in `EF_BASELINE_CUTOVER_DECISION_MATRIX.md` are met and the baseline is created. Then decide explicitly: either “freeze legacy and add only to baseline” or “add to both” (two branches of migrations) with documented process.

---

## 2. Prerequisites before creating InitialBaseline

| Prerequisite | Description |
|--------------|-------------|
| Graph integrity confirmed | `EF_MIGRATION_GRAPH_INTEGRITY_AUDIT.md` and `EF_MIGRATION_GRAPH_INTEGRITY_DECISION.md` — discoverable chain and script-only set are coherent and fully governed. |
| Full backup | Any DB used to validate baseline (e.g. dev) must be backed up before applying baseline script. |
| Environment agreement | Which environments will stay on legacy chain vs which will use baseline (e.g. new tenants, new regions, greenfield only). |
| Snapshot strategy chosen | InitialBaseline must represent one of: (a) current EF-discoverable schema only, (b) full operational schema including script-only history, or (c) current schema after an idempotent sync layer — see §3. |
| No in-place deletion of migrations | Legacy migration files (all 142) remain in repo; no rename/reorder/delete. |

---

## 3. What InitialBaseline should represent

| Option | Description | When to use |
|--------|-------------|-------------|
| **(a) Current EF-discoverable schema only** | Baseline snapshot and migration reflect only what the **95** discoverable migrations produce. Script-only (47) schema is **not** in the baseline. | When new environments do not need tables/columns that exist only in script-only migrations (e.g. CommandProcessingLogs, WorkflowInstances, etc.). Rare for CephasOps if app expects full schema. |
| **(b) Full operational schema including script-only history** | Baseline snapshot and migration represent the **entire** current operational schema: effect of 95 discoverable **plus** effect of 47 script-only (or equivalent idempotent application). New DBs get one “InitialBaseline” migration that creates everything. | **Recommended for CephasOps.** New environments get a single idempotent script that creates all tables/columns/indexes the app needs. Requires deriving “full” schema from current model or from legacy + script-only application order. |
| **(c) Current schema after an idempotent sync layer** | Same as (b) but the baseline is generated by first applying an idempotent “sync” migration to a DB that has only the discoverable chain, then capturing that state as the baseline. | When you want to avoid hand-authoring the full baseline SQL; instead you run a one-off idempotent sync (IF NOT EXISTS, etc.), then generate the baseline from that DB. |

**Recommendation for CephasOps:** **(b) Full operational schema.** The app and domain expect tables from both discoverable and script-only migrations (e.g. CommandProcessingLogs, WorkflowInstances, WorkflowSteps, JobExecutions, PayoutAnomalyReview, etc.). InitialBaseline migration should be a single, idempotent script that creates the full schema. Source of truth for “full schema” can be: (1) current `ApplicationDbContext` + configurations, or (2) concatenation of legacy idempotent scripts in dependency order. Prefer (1) with a generated script from the current model (e.g. from a temporary “clean” DbContext that has no prior migrations, or from schema export).

## 4. Existing environments (with __EFMigrationsHistory)

- **Do not** wipe or replace `__EFMigrationsHistory` for existing DBs.
- **Continue** applying the legacy chain (94 discoverable via `dotnet ef database update` or bundle) and script-only migrations via idempotent/repair scripts as today.
- **Optional later:** After all legacy migrations are applied, add a single “sync to baseline” idempotent migration or script that brings schema in line with the baseline (e.g. ADD COLUMN IF NOT EXISTS for any remaining drift) and insert one row into history for that sync. This is environment-specific and optional.
- **Rollback:** If baseline transition is postponed, existing environments are unchanged; they keep using legacy chain + scripts. No rollback of DB state is needed.

---

## 5. New environments (provisioning)

- **Option A — Legacy path:** Create DB; run idempotent script that applies all 94 + 47 (or the consolidated idempotent script from runbook); populate `__EFMigrationsHistory` with all applied migration IDs. Same as today.
- **Option B — Baseline path (after InitialBaseline exists):** Create DB; run the single InitialBaseline migration script (idempotent); insert **one** row into `__EFMigrationsHistory` with the baseline migration ID. No legacy migration IDs. New migrations (if added to baseline track) are applied after that via `dotnet ef database update` from the baseline snapshot.

---

## 6. Legacy migrations: remain in-repo or archival

- **Remain in-repo:** All 141 migration files stay in `Migrations/` for traceability, for existing DBs, and for audit. No move to archival in this strategy.
- **Optional later:** If the team adopts a “baseline-only” branch and freezes legacy, legacy files could be moved to a subfolder (e.g. `Migrations/Legacy/`) **only if** tooling and EF context are updated so that the main context no longer loads them (e.g. separate assembly or output-dir). Document fully before doing so.
- **Reference-only:** Legacy remains the source of truth for “what was applied” to existing DBs; baseline is the source of truth for new DBs.

---

## 7. CI / validator / classifier after baseline adoption

| Component | Evolution |
|-----------|-----------|
| **Validator** | If baseline track exists in the same repo (e.g. second output-dir or second context), validator may need to expect two “chains” or only validate the active track. If only legacy is extended until cutover, no change until baseline is added. When baseline exists: either (1) validate only legacy, or (2) validate both with separate expected counts. Document which chain(s) are “active.” |
| **Classifier** | Same: classify the “active” chain. If new migrations are added only to baseline, classifier may need to run against baseline folder or context. |
| **CI** | migration-hygiene.yml today runs on `Migrations/**/*.cs`. If baseline lives under `Migrations/Baseline/`, include that path; ensure CI does not double-count or fail on baseline-only files. |
| **Runbook** | Must state: when to use legacy bundle/update vs baseline script; how to add a new migration (to legacy vs baseline); which environments use which track. |

---

## 8. Cutover criteria

- Use **`docs/operations/EF_BASELINE_CUTOVER_DECISION_MATRIX.md`** for the decision matrix (continue governed model vs idempotent sync vs InitialBaseline vs full re-baseline).
- Cutover to “create InitialBaseline” when: prerequisites above are met, team has agreed on (b) or (c), and backup/rollback plan is in place.

---

## 9. Rollback if baseline transition is postponed

- **No DB rollback** needed; existing DBs are untouched.
- **Repo:** If a baseline migration and snapshot were added in a branch, do not merge; or revert the baseline commit. Legacy chain remains the only active path.
- **Docs:** Revert any “we now use baseline for new envs” wording; runbook stays “legacy + scripts only.”
