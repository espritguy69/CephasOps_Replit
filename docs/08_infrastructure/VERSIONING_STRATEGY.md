# CephasOps Versioning Strategy

**Last Updated:** 2025-01-XX  
**Status:** ✅ Active

---

## Overview

CephasOps uses **Semantic Versioning (SemVer)** with automatic version calculation based on Git history and branch naming conventions.

**Format:** `MAJOR.MINOR.PATCH[-PRERELEASE][+BUILD]`

Examples:
- `1.0.0` - Production release
- `1.1.0-alpha.5` - Development branch (5th commit)
- `1.0.1-beta.2` - Release candidate (2nd commit)
- `1.2.3+abc1234` - Build with commit hash

---

## Version Components

### MAJOR Version
Incremented when:
- Breaking API changes
- Database schema changes requiring migration
- Major feature additions that change system behavior
- Commit message contains: `+semver: breaking` or `+semver: major`

### MINOR Version
Incremented when:
- New features added (backward compatible)
- New modules added
- New API endpoints added
- Commit message contains: `+semver: feature` or `+semver: minor`

### PATCH Version
Incremented when:
- Bug fixes
- Security patches
- Performance improvements
- Commit message contains: `+semver: fix` or `+semver: patch`

---

## Branch-Based Versioning

### Main Branch
- **Tag:** None (production releases)
- **Increment:** None (manual via tags)
- **Example:** `1.0.0`, `1.1.0`, `2.0.0`

### Development Branch
- **Tag:** `alpha`
- **Increment:** Minor version
- **Example:** `1.1.0-alpha.5`, `1.2.0-alpha.12`

### Release Branches (`release/*`)
- **Tag:** `beta`
- **Increment:** None (inherits from development)
- **Example:** `1.1.0-beta.3`

### Feature Branches (`feature/*`)
- **Tag:** None
- **Increment:** Inherits from base branch
- **Example:** `1.1.0-feature-123.4`

### Hotfix Branches (`hotfix/*`)
- **Tag:** None
- **Increment:** Patch version
- **Example:** `1.0.1-hotfix-security.2`

## Automatic Version Calculation

### GitVersion Tool
CephasOps uses [GitVersion](https://gitversion.net/) to automatically calculate versions based on:
- Git tags (e.g., `v1.0.0`)
- Branch names
- Commit messages
- Merge history

### Configuration
Configuration file: `GitVersion.yml` (root directory)

### Integration
- **.NET Projects:** `Directory.Build.props` reads GitVersion variables
- **CI/CD:** GitHub Actions workflow calculates version on every build
- **Manual:** Use `scripts/bump-version.ps1` for manual version bumps

---

## Version in .NET Projects

### Directory.Build.props
All projects inherit version from `backend/Directory.Build.props`:

```xml
<Version>1.0.0</Version>
<AssemblyVersion>1.0.0.0</AssemblyVersion>
<FileVersion>1.0.0.0</FileVersion>
```

### GitVersion Integration
When GitVersion is available, versions are automatically set:
- `Version` = NuGet version (e.g., `1.1.0-alpha.5`)
- `AssemblyVersion` = Assembly version (e.g., `1.1.0.0`)
- `FileVersion` = File version (e.g., `1.1.0.5`)
- `InformationalVersion` = Full semantic version

## Manual Version Bumping

### Using PowerShell Script
```powershell
# Bump patch version
.\scripts\bump-version.ps1 -Type patch -Message "Fix critical bug" -CreateTag -Push

# Bump minor version
.\scripts\bump-version.ps1 -Type minor -Message "Add new feature" -CreateTag -Push

# Bump major version
.\scripts\bump-version.ps1 -Type major -Message "Breaking changes" -CreateTag -Push
```

### Using Git Tags
```bash
# Create version tag
git tag -a v1.0.0 -m "Release v1.0.0"

# Push tag
git push origin v1.0.0
```

### Using Commit Messages
Include semantic version hints in commit messages:
```
+semver: major - Breaking API changes
+semver: minor - New feature added
+semver: patch - Bug fix
+semver: none - No version change
```

## CI/CD Versioning

### GitHub Actions
The `.github/workflows/versioning.yml` workflow:
1. Calculates version using GitVersion
2. Builds projects with calculated version
3. Creates Git tags for main branch releases
4. Creates GitHub releases automatically

### Workflow Triggers
- **Push to main:** Creates release tag and GitHub release
- **Push to development:** Builds with alpha version
- **Manual dispatch:** Allows manual version bumping

## Release Process

### Development Release
1. Work on `development` branch
2. Commits automatically increment alpha version
3. Merge to `main` when ready

### Production Release
1. Merge `development` → `main`
2. CI/CD automatically:
   - Calculates final version
   - Creates Git tag (e.g., `v1.0.0`)
   - Creates GitHub release
   - Builds and packages artifacts

### Hotfix Release
1. Create `hotfix/*` branch from `main`
2. Fix issue
3. Merge to `main` (increments patch version)
4. Merge back to `development`

## Best Practices

1. **Always use semantic versioning** - Follow MAJOR.MINOR.PATCH format
2. **Tag releases** - Create Git tags for all production releases
3. **Use commit messages** - Include `+semver:` hints for automatic versioning
4. **Document breaking changes** - Use MAJOR version for breaking changes
5. **Keep changelog** - Document changes in release notes

## Troubleshooting

### Version Not Updating
- Check GitVersion is installed: `dotnet tool list -g`
- Verify `GitVersion.yml` configuration
- Check branch naming matches conventions

### Wrong Version Calculated
- Review GitVersion output: `dotnet-gitversion`
- Check for conflicting tags
- Verify branch naming

### CI/CD Version Issues
- Check GitHub Actions logs
- Verify GitVersion setup in workflow
- Check for permission issues with tag creation
